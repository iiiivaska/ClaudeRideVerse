# Антипаттерны
> Собрание архитектурных антипаттернов, которых следует избегать при развитии кодовой базы RideVerse.

Эти антипаттерны собраны из обоих архитектурных документов. Каждый из них нарушает одно или несколько [[Обзор#Нерушимые правила|нерушимых правил]] архитектуры.

---

## 1. God-пакет `FogRideKit`

**Проблема:** один пакет со всем сразу -- теряем преимущества модульности.

**Почему плохо:** невозможно переиспользовать отдельные части, тесты тянут всё приложение, compile-time растёт, preview становятся неподъёмными.

**Правильно:** 18 SPM-таргетов, каждый с чётко определённой ответственностью. См. [[Обзор#Сводная таблица пакетов]].

---

## 2. Feature импортирует другой feature

**Проблема:** `TripDetailFeature` тянет `RecordingFeature`, чтобы переиспользовать кнопку.

**Почему плохо:** создаёт зависимость между feature-модулями, усложняет навигацию, нарушает [[Обзор#Нерушимые правила|правило 4]].

**Правильно:** общая кнопка выносится в [[Прочие пакеты|DesignSystem]]. Навигация между фичами решается роутером на уровне App.

---

## 3. Package импортирует Domain

**Проблема:** переиспользуемый пакет (например, `LocationRecording`) импортирует `FogRideModels`, чтобы использовать тип `Trip`.

**Почему плохо:** инверсия зависимостей -- пакет становится непереносимым в другие проекты. Нарушает [[Обзор#Нерушимые правила|правило 1]] (пакеты не знают про доменные сущности).

**Правильно:** если нужен общий тип -- поднимаем его в пакет (через generics или protocol), не в Domain. Пакеты оперируют примитивами: `CLLocation`, `H3Index`, `[CLLocationCoordinate2D]`.

---

## 4. View обращается к repository напрямую

**Проблема:** SwiftUI View напрямую вызывает `TripRepository.save()`.

**Почему плохо:** View получает знание о хранилище, тестирование усложняется, бизнес-логика размазывается по UI-слою.

**Правильно:** View -> ViewModel -> Service -> Repository. Три слоя абстракции -- плата за тестируемость. См. [[Domain#View -> ViewModel -> Service]].

---

## 5. SwiftUI `@Query` для GPS-точек

**Проблема:** использование `@Query` для загрузки трека из 10k+ GPS-точек.

**Почему плохо:** `@Query` материализует весь dataset целиком в памяти. Для треков из 10k+ точек это приводит к краху (out of memory) или серьёзным тормозам.

**Правильно:** GRDB observation через async stream в ViewModel. Raw-точки хранятся в GRDB (не в SwiftData), загружаются пагинированно или стримом.

---

## 6. Domain-событие с UI-контекстом

**Проблема:** `DomainEvent.achievementUnlocked(achievement, showToast: Bool)` -- `showToast` попадает в доменное событие.

**Почему плохо:** `showToast` -- это решение UI, не Domain. Domain не должен знать, как именно UI отображает информацию. Нарушает [[Обзор#Нерушимые правила|правило 2]].

**Правильно:** `DomainEvent` содержит только доменные данные. UI-слой (Feature) сам решает, показывать toast, sheet или что-то ещё:

```swift
// Правильно:
public enum DomainEvent: Sendable {
    case tripFinished(Trip)
    case hexCountCrossed(Int)
    case maxSquareGrew(side: Int)
}
// Feature сам решает отображение при получении события.
```

---

## 7. Domain импортирует SwiftUI

**Проблема:** использование `View`, `Color`, `Image` в Domain-слое.

**Почему плохо:** Domain становится привязан к UI-фреймворку. Нарушает [[Обзор#Нерушимые правила|правило 2]] (Domain не знает про UI).

**Правильно:** `@Observable` в Domain -- нормально (Observation framework -- Foundation-адъюнкт). Но никаких `View`, `Color`, `Image`. Computed properties вроде `trip.thumbnail: UIImage` -- запрещены в моделях.

---

## 8. UIKit в Domain-моделях

**Проблема:** `trip.thumbnail: UIImage { ... }` как computed property на `@Model`.

**Почему плохо:** UIKit-зависимость в чистой доменной модели.

**Правильно:** в модели -- только pure computed properties (`trip.elevationGainFeet: Double { elevationGainMeters * 3.28 }`). Рендеринг thumbnail -- обязанность сервиса или feature-слоя.

---

## Быстрая проверка

При добавлении нового кода задай себе вопросы:

1. **Можно ли этот пакет перенести в другой проект без FogRide?** Если нет -- возможно, в него проникла доменная логика (антипаттерн 3).
2. **Знает ли этот пакет, как будет выглядеть UI?** Если да -- нарушение границ (антипаттерн 7).
3. **Тянет ли этот feature другой feature?** Если да -- выноси общее в [[Прочие пакеты|DesignSystem]] (антипаттерн 2).
4. **Обращается ли View напрямую к хранилищу?** Если да -- нужен промежуточный сервис (антипаттерн 4).
5. **Используется ли `@Query` для больших датасетов?** Если > 1k записей -- переходи на GRDB observation (антипаттерн 5).
