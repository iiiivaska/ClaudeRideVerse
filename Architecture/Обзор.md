# Обзор архитектуры
> Философия модульности, золотое правило, трёхслойная структура, сводная таблица 18 пакетов, стратегии хранения и тестирования, порядок разработки.

## Философия деления

**Главный принцип:** пакеты не знают о существовании FogRide. Они знают только о предметной области: «карты», «GPS», «гексы», «сенсоры». Это даёт два эффекта:
1. Пакеты переиспользуемы в будущих приложениях.
2. Бизнес-логика приложения не может случайно проникнуть в библиотеку через задние ворота.

## Нерушимые правила

1. **Пакеты не знают про доменные сущности.** `Trip`, `User`, `Ride`, `Achievement` -- запрещены в пакетах. Они оперируют примитивами: `CLLocation`, `H3Index`, `[CLLocationCoordinate2D]`, `TrackPoint` (структура пакета, не FogRide).
2. **Пакеты не знают про UI приложения.** Могут предоставлять SwiftUI-вью (карта, HUD-компоненты), но стилизация -- извне через environment/modifier, без захардкоженных цветов FogRide.
3. **Направление зависимостей строго сверху вниз.** Никаких обратных стрелок, никаких циклов.
4. **Один feature-модуль не импортирует другой feature-модуль.** Навигация между фичами решается роутером на уровне App.

## Трёхслойная структура

```
┌─────────────────────────────────────────────────────────┐
│  App: FogRideApp                                        │
│  -- composition root, DI, deep links, app lifecycle     │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Features/*                                             │
│  -- SwiftUI Views + feature-specific @Observable models │
│  -- знают про Domain, могут тянуть Packages для UI      │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Domain/                                                │
│  ├ FogRideModels       -- Trip, Bike, Achievement, etc. │
│  ├ FogRidePersistence  -- SwiftData + GRDB хранилища    │
│  └ FogRideServices     -- TripRecorder, FogCalculator   │
│  Знает про Packages, НЕ знает про UI и про Features     │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Packages/*  (MapCore, HexCore, LocationRecording, ...) │
│  Не знают ни про Domain, ни про Features, ни про App    │
└─────────────────────────────────────────────────────────┘
```

## Стратегия хранения

Двойная стратегия: **SwiftData + CloudKit** для метаданных (Trip, Bike, Achievement, UserStats) и **GRDB** для тяжёлых данных (GPS-точки, hex-снапшоты).

- SwiftData синхронизирует через CloudKit -- метаданные поездок доступны на всех устройствах.
- GRDB хранит raw-точки треков (10k+ на поездку) и compacted hex-снапшоты -- не синхронизируются, слишком тяжело.
- `TripRepository` инкапсулирует оба хранилища за единым protocol-интерфейсом.

Подробнее: [[Domain#Domain/FogRidePersistence]]

## Стратегия H3 resolution

Дефолтный resolution -- **r9** (~174 м edge). Adaptive compaction для рендеринга по zoom:

| Zoom карты | Detail resolution | Метод |
|---|---|---|
| z >= 13 | r9 (native) | Прямые ячейки |
| z 11-12 | r8 | `lossyCompact` с threshold 0.5 |
| z 9-10 | r7 | `lossyCompact` |
| z <= 8 | r6 | Грубые полигоны |

Подробнее: [[HexKit#Adaptive compaction для рендеринга]]

## Стратегия тестирования

- **Unit-тесты** -- для каждого пакета независимо, с fixture-данными.
- **Property-based тесты** -- для HexCore (parent/children round-trip, pentagon handling).
- **Snapshot-тесты** -- для MapCore, MapFogOfWar (рендер карты с fixture-стилем).
- **Performance benchmarks** -- FPS для fog rendering (target >= 55 FPS на iPhone 14), multipolygon rebuild (target: 10k < 50ms).
- **Mock-based тесты** -- TripRecorder с mock-репозиторием, без реальной SwiftData/GRDB.

## Порядок разработки

Строим снизу вверх, чтобы каждый пакет можно было немедленно проверить:

1. **[[HexKit|HexCore + HexGeometry]]** (1-2 недели) -- легко тестируются unit-тестами, без UI.
2. **[[LocationKit|LocationRecording + LocationFiltering + LocationAnalysis]]** (2-3 недели) -- тестируются с записанными фикстурами CLLocation.
3. **[[MapVerse|MapCore + MapOverlays]]** (2 недели) -- первый визуальный milestone: карта с треком.
4. **[[MapVerse|MapFogOfWar]]** (1-2 недели) -- соединяет HexGeometry и MapCore, кульминация фазы 0.
5. **LocationMotion, MapOfflineRegions, WorkoutKit, BLESensorCore, CyclingSensors, TrackExport** -- по мере надобности в фазах 1-5.

## Масштабирование на новые фичи

**Apple Watch companion (фаза 5):** переиспользуем `LocationRecording`, `LocationFiltering`, `HexCore`, `HexGeometry`, `CyclingSensors`, `WorkoutKit`. Новый feature-модуль `WatchCompanion` в отдельном watchOS-таргете. Domain -- тот же, общий через `WCSession`.

**Публичный heatmap (фаза 3):** новый пакет `HeatmapKit`, feature-модуль `CommunityHeatmapFeature`, доменный сервис `HeatmapService`. Возможно -- сетевой слой `FogRideBackend` в Domain.

**Отдельное walking-приложение:** новый App-таргет. Переиспользуем все Packages. НЕ переиспользуем `Domain/*` и `Features/*` -- у них своя доменная модель. Это и есть главный бонус от дисциплинированного деления.
